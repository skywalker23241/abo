# 影子

今天早上闲来无事，试了下之前注册 ChatGPT Plus 的渠道，发现又可以使用了，五一可以爽用了，然后乘着这个好兴致，作一篇博客来分享下自己最近~~白嫖~~使用AI、Agent的渠道，恐怕以后这些渠道会被封杀，作此博客留念！

# 雨后春笋

最近 AI 工具真的越来越多了，真的是如雨后春笋一般，多到我有时候已经不太想去分辨“哪个模型最强”这种问题了。

像这两天发布的 GPT 5.5 和 GPT IMAGE-2 两个王炸，简直比之前 Gemini 的 Nano Banana 还要更加惊艳四座！

但是真正用起来以后会发现，模型能力当然重要，但入口/渠道也很重要。你是在网页里问一句话，还是在 IDE 里改一个项目，还是在终端里让它读代码、跑命令、定位问题，这些体验完全不是一回事。

之前我会更关注模型本身，比如 GPT、Claude、Gemini 谁更聪明，谁写代码更稳，谁生成图片更像真人。但现在我更关心的是：**我在什么情况下会自然地打开它？它能不能接住我当下的任务？**

所以这篇就简单记录一下我最近常用的几个 AI 渠道：ChatGPT Plus、Codex、Antigravity，以及 Claude Code。不是严肃评测，也不是推荐榜单，只是我这一段时间的真实使用感受。


# GPT/Codex

**GPT** 现在对我来说更像是一个“默认入口”。

很多时候我不是为了完成一个特别复杂的任务才打开它，而是遇到什么问题就先丢进去问一下。比如整理一段想法、解释一个概念、帮我润色文字、拆解一件事情的步骤，或者单纯拿来做搜索前的预处理，又或是回答我的一些小问题。

它最大的优势不是某一项能力特别离谱，而是综合能力很均衡。文字、图片、代码、资料整理、日常问答都能做一点，而且使用门槛很低。你不需要先打开项目，也不需要配置什么环境，直接开聊就行。

这点其实很重要。很多工具理论上很强，但如果每次使用前都要先切环境、配上下文、想清楚怎么发指令，那它就很难成为日常工具。GPT 的优势就在于它足够顺手，像一个随时能接话的入口。

当然，它也不是所有场景都最合适。比如真的要改一个本地项目，或者要连续读很多文件、执行命令、反复验证结果，它就没有专门的开发工具那么自然。但如果只是想先把问题想清楚，或者快速得到一个方向，它依然是我最常打开的 AI（双端）。

我自己也是有多个 GPT 的账号，以供自己日常使用，因为都是普号，生图和 Codex 的额度都不算多，只能凑合用，不过有时能要到饭，就可以爽用！

**Codex** 给我的感觉就不太一样了，它更像是一个可以直接动手的开发伙伴。

以前我用 AI 写代码，很多时候还是复制粘贴式的：我把代码贴给它，它给我一段建议，我再自己拿回来改。这个过程当然有用，但中间隔了一层，人还是要不断搬运上下文。

Codex 的体验更接近“把它放进项目里”。它可以读仓库、理解文件结构、执行命令、跑检查，然后根据实际结果继续调整。对我这种经常折腾博客、配置、前端页面和各种小项目的人来说，这种体验就很舒服。

比如我要改一篇文章、调整一个组件、排查一个构建错误，Codex 不只是给我一个抽象建议，而是能先看项目现在长什么样，再给出具体修改。它的价值不只是“写代码”，而是把读项目、改文件、验证结果这几个步骤串起来。

我现在对 Codex 的期待也比较明确：它适合处理有上下文的工作。只要任务和本地仓库有关，我就更愿意让它先看代码，而不是自己凭记忆描述一大堆。

当然，这类工具也要求你自己保持清醒。AI 可以加快动手速度，但最终要不要这样改、会不会影响别的地方、文章语气是不是还是自己的，这些判断还是要人来做。它像副驾驶，但方向盘不能完全丢出去。

# Antigravity

Antigravity 我之前在 Gemini 3 那篇里已经简单提过一次。

::link{url="https://jackcooper.qzz.io/new-ai-thought/"}

它吸引我的地方主要是 IDE 形态和浏览器联动。现在很多 AI 工具都在往 **“编辑器 + Agent + 浏览器”** 的方向走，Antigravity 算是这个趋势里比较有代表性的一个。

但是最近谷歌对于 Antigravity 的投入还是算少的吧，还有之前背着用户偷偷改额度的规则，5h重置变成5d，引起 Pro 用户众怒。

对我来说，它现在也就 Claude Opus 4.6 的那个模型和 Gemini 3 Pro 聪明一点，用起来额度掉的也快，其他方面基本上是废了。

不过值得关注的是，最近谷歌在 GAS 里面投入的精力变多了，之前是只支持 API Key 使用，现在支持账户链接了，看下以后那个玩意的效果咋样。

# Claude Code

Claude Code（CC） 则是另一个很典型的开发入口，基本上高级一点的程序员都用过这玩意。

Claude 系列模型给我的印象一直是上下文理解比较稳，写代码的时候也比较有工程感。

Claude Code 把这种能力放到终端和项目环境里以后，就更适合处理那些需要仔细读代码、理解结构、再动手修改的任务。

我会把它和 Codex 放在相近的位置上看，但两者的气质不完全一样。Codex 更像是和当前项目一起工作的本地协作者，而 Claude Code 给我的感觉更偏“冷静地读完，然后给出比较成体系的修改方案”。

当然，这不是说谁一定更强。现在这些工具更新太快了，今天的结论过几个月可能就变了。我更愿意按任务来选择：如果我想让 AI 帮我快速进入一个代码库、理解一段复杂逻辑，Claude Code 是一个很值得尝试的入口。

尤其是一些不是单点代码补全，而是涉及多个文件、多个模块、需要保持风格一致的工作，Claude Code 的价值会更明显。它不是那种只适合问答的 AI，而是更像一个能在工程现场参与讨论的人。

我目前用的渠道是 Any 大善人的渠道，不过最近总是 “503”，不知道啥时候能再蹬上，u1s1，CC是真的强，感觉一次直出的效果比其他工具好多了！

# 我现在怎么分配它们

如果简单总结一下我现在的使用方式，大概是这样：

- **GPT/Codex**：默认入口。适合日常问答、写作、整理想法、解释概念、快速获得方向。

- **Antigravity**：IDE 和前端体验入口。适合观察 Gemini 的能力、尝试页面开发、体验浏览器联动和新型 AI IDE。
- 
- **Claude Code**：代码推理入口。适合复杂代码理解、重构思路、工程化修改，以及需要稳一点的代码分析场景。

这样分下来以后，我反而不太纠结哪个工具一定最好了。因为它们对应的不是同一个场景。

有时候我只是想问一句“这个概念怎么理解”，那打开 GPT 就够了；有时候我需要改博客里的一个页面，那 Codex 更顺手；有时候我想看看 AI IDE 未来会变成什么样，就去试 Antigravity；如果碰到比较复杂的代码结构，再让 Claude Code 参与进来。

工具多了以后，真正重要的不是把每个工具都用满，而是知道什么时候该用哪个。否则很容易陷入一种状态：每天都在试新工具，但真正完成的事情并没有变多，蹬完一个的额度，再去蹬另一个，这就是我用AI的方法。

# 后记

这几年 AI 的入口变化其实挺明显的。

一开始大家印象里的 AI 可能就是网页聊天框，你问一句，它答一句。后来变成插件、API、浏览器扩展，再后来进入 IDE、终端、操作系统和各种工作流。

也就是说，AI 正在从一个“你主动去问的工具”，慢慢变成一个“嵌在你工作现场里的助手”。这个变化我觉得比单个模型分数高一点低一点更值得关注。

对我来说，现在最重要的不是追每一个新工具，而是找到一套自己顺手的组合。ChatGPT Plus 负责日常入口，Codex 负责本地项目，Antigravity 用来观察新 IDE 形态，Claude Code 作为代码推理和工程分析的补充。

可能过一段时间这个组合又会变，毕竟 AI 发展速度太快了。但至少在现在，这几个渠道基本覆盖了我大部分使用场景。
